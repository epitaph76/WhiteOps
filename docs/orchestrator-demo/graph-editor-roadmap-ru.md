# Roadmap: Визуальный Оркестратор (Miro-подобная схема)

## Целевой сценарий

Пользователь собирает оркестрацию на схеме:

- справа есть палитра элементов (Codex, Qwen, будущие модели/роли);
- элементы перетаскиваются на канвас;
- между элементами рисуются стрелки (подчинение/передача задач);
- по клику на узел открывается диалог:
  - у менеджера (Codex) видно, какие задачи кому назначены и текущий прогресс;
  - у воркеров (Qwen) видно свою очередь задач, статус, логи и результат.

- [ ] Реализовать перемещение узлов по канвасу.
- [ ] Реализовать рисование связей между узлами:

  - [ ] тип связи (например: `manager_to_worker`, `peer`, `feedback`);

- [ ] Добавить модель состояния схемы:
  - [ ] `nodes` (id, type, label, position, config);
  - [ ] `edges` (id, fromNodeId, toNodeId, relationType);
  - [ ] версионность схемы (revision).
- [ ] Добавить инспектор узла:
  - [ ] свойства узла (агент, таймауты, cwd, лимиты);

- [ ] Добавить модальное окно диалога менеджера:
  - [ ] чат с менеджером;

  - [ ] статус исполнения по воркерам.
- [ ] Добавить модальное окно диалога воркера:
  - [ ] входная задача;
  - [ ] поток логов;
  - [ ] финальный ответ/артефакты.
- [ ] Добавить realtime-обновление по WebSocket/SSE для узлов и связей.
- [ ] Добавить сохранение/загрузку схемы (локально и через backend API).
- [ ] Добавить UX-детали:
  - [ ] миникарта канваса;
  - [ ] zoom/pan;
  - [ ] snap to grid;
  - [ ] мультивыбор и горячие клавиши.

## Минимальный MVP (рекомендуемый порядок)

- [ ] Этап 1: Канвас + узлы без связей, сохранение схемы.
- [ ] Этап 2: Связи + валидация графа + запуск по графу.
- [ ] Этап 3: Realtime статусы узлов и логи.
- [ ] Этап 4: Диалоговые окна менеджера и воркеров.
- [ ] Этап 5: Улучшение UX (масштаб, миникарта, хоткеи, мультивыбор).

## Важные технические решения заранее

- [x] Протокол realtime: оставили SSE (`/graph-runs/:runId/events`, `/tasks/:taskId/events`).
- [x] Источник правды по схеме: backend-first (сейчас in-memory хранилище orchestrator).
- [x] Формат trace менеджера: структурированные события, а не сырой текст.
- [x] Ограничения исполнения: таймауты, лимиты параллелизма, отмена на уровне узла/графа.

## Текущие ограничения Backend

- [ ] Персистентность: хранилище пока in-memory, после рестарта данные графов/ранов теряются.
- [ ] Пагинация/фильтрация: базовая, без сложных query-моделей.
- [ ] Аудит/безопасность: есть token auth + ACL, но без внешнего IAM/SSO.
