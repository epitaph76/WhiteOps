# WhiteOps

WhiteOps - это система оркестрации AI-помощников для разработки на Flutter. Продукт объединяет Flutter-клиент для desktop/mobile и backend, который координирует несколько coding assistant-агентов, задаёт уточняющие вопросы, декомпозирует задачу на подзадачи, запускает специализированных агентов, валидирует результат и улучшает процесс на основе пользовательского фидбэка.

## Видение продукта

Ключевая идея - не один чат-бот, а целая AI-команда:

- один управляющий агент получает задачу, уточняет требования и строит план;
- несколько рабочих агентов выполняют специализированные роли: написание кода, тестирование, отладка, рефакторинг, ревью;
- пользователь наблюдает за процессом и управляет им из Flutter-приложения на компьютере или телефоне;
- система сохраняет ошибки, замечания, слабые места и использует их для улучшения дальнейшей оркестрации.

Первичная целевая область - Flutter. WhiteOps должен помогать создавать фичи, исправлять баги, генерировать тесты, улучшать архитектуру и координировать многошаговые задачи разработки между несколькими помощниками.

## Базовая концепция

WhiteOps состоит из двух частей:

1. Frontend
   Flutter-клиент для создания задач, просмотра прогресса, подтверждения результатов, ревью и управления с телефона.

2. Backend
   Центральный сервер оркестрации, отвечающий за планирование, выполнение агентами, логи, память, валидацию и интеграцию с провайдерами моделей.

Backend - это мозг системы. Flutter-приложение - это операторская панель.

## Основной сценарий использования

Пользователь даёт задачу вроде:

`Создай Flutter login flow с валидацией, интеграцией API и тестами.`

После этого WhiteOps должен:

1. задавать уточняющие вопросы, пока задача не станет достаточно конкретной;
2. определить критерии готовности;
3. разбить работу на подзадачи;
4. распределить подзадачи между специализированными агентами;
5. собрать результаты в виде патчей, заметок, тестов и логов;
6. провалидировать результат инструментами;
7. показать итог пользователю на подтверждение;
8. сохранить пользовательский фидбэк и использовать его в следующих запусках.

## Модель агентов

WhiteOps использует multi-agent архитектуру.

### Управляющий агент

Ответственность:

- собирает требования;
- задаёт уточняющие вопросы;
- строит план выполнения;
- распределяет задачи между рабочими агентами;
- решает, нужно ли продолжать, повторить, эскалировать или снова спросить пользователя;
- анализирует неудачи и обновляет стратегию оркестрации.

В самом раннем прототипе эту роль может выполнять Codex.

### Рабочие агенты

Примеры ролей:

- `Coder Agent`
- `Test Agent`
- `Debug Agent`
- `Refactor Agent`
- `Reviewer Agent`
- `Docs Agent`

На раннем этапе локального тестирования рабочие агенты могут работать через отдельные CLI-сессии, например несколько окон `Qwen Code`. Позже тот же слой оркестрации должен поддерживать полноценные API-провайдеры.

## Обзор архитектуры

```text
Flutter Client (desktop / mobile)
            |
            v
      API + WebSocket
            |
            v
     WhiteOps Backend
            |
            +-- Task Manager
            +-- Orchestrator
            +-- Agent Runtime
            +-- Validation Tools
            +-- Memory / Feedback Store
            |
            +-- Provider Adapters
                 +-- CLI bridge (dev)
                 +-- API providers (prod)
                 +-- Local models
```

## Зона ответственности frontend

Flutter-клиент должен оставаться тонким. Его задача - дать удобную панель управления:

- создавать задачи и управлять ими;
- вести диалог с управляющим агентом;
- показывать стадии выполнения и статусы агентов;
- отображать логи, патчи, diff и результаты валидации;
- подтверждать или отклонять решения;
- отправлять запросы на доработку с компьютера или телефона.

Клиент не должен содержать в себе логику оркестрации.

## Зона ответственности backend

Backend должен владеть всей ключевой логикой:

- жизненный цикл задач и state machine;
- оркестрация и делегирование;
- абстракция над провайдерами моделей;
- история выполнения и логи;
- хранение артефактов;
- автоматическая валидация;
- память и обработка фидбэка;
- аутентификация, биллинг и контроль доступа.

## Сетевая модель

WhiteOps строится вокруг центрального сервера со стабильным адресом.

- Flutter-клиенты подключаются к backend по HTTPS и WebSocket.
- Машины разработчиков могут выступать воркерами и сами подключаться к серверу через VPN или туннель.
- Это снимает зависимость от статического публичного IP у рабочей машины.

Такая схема позволяет удобно управлять оркестром с телефона, даже если основной компьютер находится за динамическим IP.

## Цикл обучения

WhiteOps не должен напрямую "обучать модель". Первая практическая форма самоулучшения - это обучение самой оркестрации.

После каждого запуска система сохраняет:

- какая задача была поставлена;
- какие агенты участвовали;
- что не получилось или было упущено;
- что исправил пользователь;
- что поймала автоматическая валидация;
- какие промпты или планы дали слабый результат.

Эти данные используются для улучшения следующих запусков через:

- более точные уточняющие вопросы;
- более сильные чек-листы планирования;
- лучшую маршрутизацию задач к нужным агентам;
- дополнительную валидацию для слабых зон;
- адаптацию промптов на основе повторяющихся ошибок.

Это policy learning, а не fine-tuning модели.

## Стратегия провайдеров

WhiteOps должен поддерживать несколько режимов выполнения через единый слой адаптеров.

### Режим разработки

Для локального прототипирования backend может отправлять запросы в несколько уже открытых CLI-помощников. Это полезно для тестирования логики оркестрации без постоянных расходов на API во время каждой итерации.

Примеры:

- одна управляющая сессия;
- несколько рабочих CLI-сессий;
- локальный прокси, который отправляет промпты и считывает ответы.

### Продакшн-режим

Для реального использования продукта backend должен работать через поддерживаемые LLM API или локально хостируемые модели. Продакшн не должен зависеть от вручную открытых CLI-окон.

## Модель подписки

Предполагаемая коммерческая модель - централизованное управление доступом и расходами через backend:

- пользователи платят WhiteOps за доступ к продукту;
- WhiteOps оплачивает использование underlying models;
- биллинг, квоты и лимиты контролируются централизованно.

Важное ограничение:

- потребительская подписка вроде `ChatGPT Plus` не эквивалентна API-доступу для стороннего продакшн-приложения;
- поэтому WhiteOps должен опираться на API-провайдеров, self-hosted модели или явно переданные пользователем учётные данные/ключи провайдера.

## Технический стек MVP

### Короткое решение

Для MVP основной backend WhiteOps делаем на `TypeScript`, а не на Python, Rust или Dart.

Основной стек:

- `Node.js + TypeScript`
- `NestJS`
- `Fastify`
- `WebSocket`
- `PostgreSQL`
- `Prisma`
- `Redis + BullMQ`
- `Docker Compose`

Для debug-режима прослойку, которая пересылает запросы в локальные CLI-клиенты, тоже делаем на `TypeScript/Node.js`, но отдельным сервисом.

### Почему основной backend именно на TypeScript

WhiteOps - это orchestration backend, а не ML-ядро. Основная нагрузка здесь такая:

- сетевой I/O;
- WebSocket-соединения;
- очереди задач;
- стриминг логов и статусов;
- запуск процессов и CLI;
- хранение состояния;
- контроль агентов и подзадач.

Для такого типа системы `TypeScript` даёт лучший компромисс между скоростью разработки, типобезопасностью и зрелостью backend-экосистемы.

Коротко по альтернативам:

- `Python` полезен для eval, embeddings, RAG и offline-обработки, но не должен быть ядром MVP;
- `Rust` хорош для performance-critical частей, но замедлит первый продуктовый цикл;
- `Dart` удобен рядом с Flutter, но слабее именно в backend orchestration и CLI/PTY-интеграциях.

Если позже понадобится Python, его добавляем как отдельный специализированный сервис, а не как замену главному backend.

### Основной backend-стек

#### Язык и runtime

- `Node.js`
- `TypeScript`

Это базовый слой для API, оркестрации, очередей, realtime и внешних интеграций.

#### Backend framework

- `NestJS`

Нужен из-за модульной архитектуры: API, orchestration, workers, gateways, auth и provider adapters удобнее держать в явных модулях, а не в разросшемся хаотичном сервисе.

#### HTTP API

- `Fastify` через адаптер NestJS

Используется как основной HTTP-слой для REST API.

#### Realtime

- `WebSocket`

Нужен для:

- живых статусов задач;
- диалога с управляющим агентом;
- стрима логов и событий;
- удалённого управления с телефона.

Для MVP предпочтительнее обычный WebSocket-протокол, без лишнего усложнения.

#### База данных

- `PostgreSQL`

В ней храним:

- пользователей;
- проекты;
- задачи;
- подзадачи;
- agent runs;
- feedback;
- weakness profiles;
- ссылки на артефакты.

#### ORM

- `Prisma`

Используется для схемы БД, миграций и типизированного доступа к данным.

#### Очереди и фоновые задачи

- `Redis`
- `BullMQ`

Нужны для:

- очереди задач агентов;
- retries;
- delayed jobs;
- параллельных worker-процессов;
- parent/child flow для подзадач оркестра.

#### Хранение артефактов

Для MVP:

- локальная файловая система или Docker volume

Позже:

- `S3` или `MinIO`

Артефакты:

- diff;
- patch;
- логи;
- результаты тестов;
- архивы проекта;
- экспортированные отчёты.

#### Аутентификация

Для MVP:

- `JWT access token`
- `JWT refresh token`

Без сложной биллинговой модели на первом этапе.

#### Логирование и наблюдаемость

Для MVP:

- структурированные логи;
- `request_id`;
- `task_id`;
- `agent_run_id`.

Позже можно добавить OpenTelemetry и централизованный сбор логов.

#### Локальная инфраструктура разработки

- `Docker Compose`

Через него поднимаем:

- `PostgreSQL`
- `Redis`
- вспомогательные сервисы, если понадобятся позже

### Прослойка для debug-режима

#### Решение

CLI bridge делаем как отдельный сервис:

- язык: `TypeScript`
- runtime: `Node.js`
- работа с псевдотерминалами: `node-pty`

Его нельзя смешивать с публичным backend API в одном процессе.

#### Зачем отдельный сервис

Причины две:

- безопасность: интерактивные CLI-процессы не должны жить внутри internet-facing API;
- изоляция отказов: зависшая CLI-сессия не должна ронять основной backend.

#### Где он работает

Для локальной разработки CLI bridge должен жить на машине разработчика, рядом с локальными CLI-клиентами, и сам подключаться к центральному backend.

Это хорошо подходит под схему с VPN/туннелем и динамическим IP.

#### Что делает CLI bridge

Он должен:

- поднимать и держать несколько CLI-сессий;
- знать, какая сессия свободна, а какая занята;
- отправлять prompt в нужную сессию;
- стримить ответ обратно в backend;
- перезапускать зависшие сессии;
- ограничивать таймауты и параллелизм;
- писать технические логи.

#### Протокол между backend и CLI bridge

Для MVP:

- `WebSocket`

Этого достаточно для двустороннего realtime-обмена, live-логов и стриминга ответов. Если позже понадобится более строгий межсервисный контракт, тогда можно перейти на `gRPC`.

#### Схема debug-режима

```text
Flutter Client
      |
      v
WhiteOps Backend
      |
      v
CLI Bridge Service
      |
      +-- CLI Session 1
      +-- CLI Session 2
      +-- CLI Session 3
      +-- CLI Session 4
```

### Что не используем как ядро оркестрации

На старте не делаем ядро системы на готовых агентных фреймворках:

- `LangGraph`
- `CrewAI`
- `AutoGen`

Коротко, что это:

- `LangGraph` - framework для агентных графов и stateful workflow вокруг LLM;
- `CrewAI` - framework для multi-agent сценариев с ролями и кооперацией агентов;
- `AutoGen` - framework для агентных диалогов и координации нескольких LLM-агентов.

Это полезные инструменты для экспериментов, но не лучший фундамент для WhiteOps MVP, потому что тебе нужен не generic agent-demo, а контролируемый backend со своей state machine, очередями, логами, артефактами и feedback loop.

Поэтому ядро WhiteOps лучше реализовать вручную:

- task state machine;
- manager workflow;
- dispatch в worker-агенты;
- provider adapter layer;
- feedback pipeline.

## Начальный MVP

Первый MVP должен быть узким по объёму, но рабочим.

### Объём MVP

- фокус на сценариях Flutter-разработки;
- один backend-сервис;
- один Flutter-клиент;
- один управляющий агент;
- от 2 до 4 рабочих агентов;
- логи, статусы и поток подтверждения результата;
- сбор фидбэка после каждого запуска.

### Поток MVP

1. Пользователь создаёт задачу в Flutter-клиенте.
2. Управляющий агент задаёт уточняющие вопросы.
3. Backend сохраняет финальные требования и критерии готовности.
4. Управляющий агент создаёт подзадачи.
5. Рабочие агенты выполняют кодинг, тестирование и ревью.
6. Backend запускает валидации.
7. Пользователь проверяет результат и отправляет подтверждение или замечания.
8. Backend сохраняет сигналы о слабых местах для будущей оркестрации.

### Рекомендуемый первый набор агентов

- `Planner / Manager`
- `Coder`
- `Tester`
- `Reviewer`

## Что не входит в MVP

Чтобы первая версия оставалась реалистичной, на старте не стоит делать приоритетом:

- полную симуляцию автономной AI-компании;
- сложные пайплайны fine-tuning;
- широкую поддержку IDE с первого дня;
- крупные командные функции совместной работы;
- сложную биллинговую систему до того, как заработает сам цикл оркестрации.

## Принципы проектирования

- Backend-first архитектура.
- Тонкий Flutter-клиент.
- Явные состояния задач вместо одного свободного чата.
- Структурированные ответы агентов, а не только свободный текст.
- Заменяемый provider layer.
- Улучшение на основе фидбэка.
- Мобильный контроль как базовая возможность, а не второстепенная функция.

## Долгосрочное развитие

Если MVP покажет себя рабочим, WhiteOps может развиться в:

- полноценную AI-команду для Flutter-проектов;
- гибридную локально-облачную платформу оркестрации;
- рабочее пространство разработчика с историей задач, памятью по проекту и переиспользуемыми паттернами выполнения;
- IDE-интегрированную систему кодинга с удалённым управлением с мобильных устройств.

## Краткое резюме

WhiteOps - это backend-центричная платформа оркестрации AI-помощников для программирования с фокусом на Flutter. Управляющий агент уточняет задачи, декомпозирует работу, координирует рабочих агентов, валидирует результат и учится на пользовательском фидбэке. Flutter frontend выступает как живая панель управления для desktop и mobile, а backend остаётся достаточно гибким, чтобы на этапе разработки работать через CLI-прототипы, а в production - через API или self-hosted модели.

## Статус реализации (MVP)

Сделан первый рабочий каркас debug-прослойки:

- путь: `services/cli-bridge`
- стек: `Node.js + TypeScript + Fastify + node-pty`
- поддержка выбора CLI-агента: `codex` или `qwen`
- API для запуска и управления сессиями: `/agents`, `/sessions`, `/sessions/:id/prompt`, `/runs`

Детали запуска и env-конфиг находятся в `services/cli-bridge/README.md`.
